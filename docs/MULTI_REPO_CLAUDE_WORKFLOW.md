# Multi-Repo Claude Workflow — gcp3 + gcp3-mobile

**Date:** 2026-04-21
**Scope:** Practical playbook for developing the gcp3 finance app across two GitHub repos with Claude Code.

---

## The problem this doc solves

The app spans two repos with separate GitHub remotes:

| Repo | Remote | Role |
|---|---|---|
| `gcp3` | `adamaslan/gcp3.git` | Python FastAPI backend + Next.js web frontend |
| `gcp3-mobile` | `adamaslan/gcp-expo1.git` | Expo/React Native mobile app |

They share a **contract** (the JSON the backend emits and both frontends consume), but nothing enforces that contract. Drift between the three surfaces is the dominant failure mode — see [SECTION_AB_IMPLEMENTATION_SUMMARY.md](./SECTION_AB_IMPLEMENTATION_SUMMARY.md): backend Section A/B work shipped, mobile components were written against a schema path that doesn't exist, web was never updated at all.

This doc is about preventing that class of failure.

---

## 1. How to think about the two repos

**The backend is the source of truth. The frontends are followers.**

Every concept (a `SignalOutput`, a `TimeframeMatrix`, an endpoint shape) has exactly one authoritative definition, and it lives in `gcp3/backend/`. Both frontends are downstream consumers. When you feel the urge to "just add it to the mobile app for now," stop — you're creating drift.

**Work loop for a cross-repo feature:**

1. Define the contract in the backend (Pydantic schema or endpoint). Ship it.
2. Regenerate the TypeScript types for both frontends from that contract.
3. Build the web UI against the new types.
4. Build the mobile UI against the same types.
5. Both frontends converge on the same backend version.

**Never do step 3 or 4 before step 1.** The Section A/B miss happened because mobile components were written speculatively before the TS types existed.

## 2. How to run Claude across both repos

### Option A (recommended): Two separate Claude sessions

Open one Claude Code session in each repo root. Each session's context is tuned to that repo's stack:
- `gcp3/` session knows Python, FastAPI, mamba, Firestore, Vercel
- `gcp3-mobile/` session knows Expo, React Native, Clerk, Reanimated

The two sessions can't see each other. You — the human — are the bridge. Use the "handoff doc" pattern in §3.

### Option B: One Claude session spanning both

Open Claude in `/Users/adamaslan/code/` (the parent). Works, but context gets muddled — Claude will conflate Next.js patterns with React Native patterns, or try to run `mamba` commands in the mobile repo. Only use this for cross-cutting tasks (e.g. "generate TS types from the Python schema and copy to both frontends").

**Rule of thumb:** default to Option A. Switch to Option B only when the task explicitly crosses the boundary.

### Model selection

- **Opus 4.7** — architecture decisions, cross-repo refactors, debugging anything that involves the contract boundary. Use when the question is "what should we do?"
- **Sonnet 4.6** — most implementation work. Use when the question is "do this specific thing."
- **Haiku 4.5** — throwaway scripts, log analysis, answering "where is X defined?" Use when the question has an obvious answer and you just want it fast.

Default to Sonnet. Escalate to Opus only when you notice Sonnet looping or making shallow suggestions.

## 3. The handoff doc pattern

When a backend change requires a frontend change, don't try to remember the delta across sessions. Write a short handoff doc in the backend repo:

```
gcp3/docs/handoff/2026-04-21_signal_output_v2.md
```

Contents:
- What changed (new field? renamed? new endpoint?)
- The new TS type (paste it)
- What each frontend needs to do
- How to verify

Then in the mobile/web Claude session, point Claude at that file. Claude reads the handoff, does the follower work, and the doc becomes the audit trail. No guessing, no drift.

## 4. Contract-first development — concrete mechanics

The contract is the Pydantic models in `gcp3/backend/schemas/`. Everything else regenerates from them.

**Tooling chain:**
1. `datamodel-code-generator` → emits OpenAPI from Pydantic
2. `openapi-typescript` → emits `.d.ts` from OpenAPI
3. Copy the generated types into `gcp3/frontend/src/types/` and `gcp3-mobile/lib/schemas/`

**Rules:**
- Hand-maintained TS mirrors of Python types are banned. If you catch one, delete it and regenerate.
- Generated files have a header `// AUTOGENERATED — do not edit` and are committed to each repo.
- A backend PR that changes a schema is incomplete until the regeneration step has been run and the two follower PRs are open.

This is item #1 in the Section A/B simplification list. It's load-bearing — do it before the next feature.

## 5. Branch, commit, PR discipline across two remotes

Each repo has its own remote, its own PR queue, its own CI. Don't let them get out of sync.

**Pattern for a cross-repo feature:**

1. Create three branches with the same name across all three surfaces:
   - `gcp3/feature/signal-output-v2` (backend)
   - `gcp3/feature/signal-output-v2` (web frontend — same repo, same branch)
   - `gcp3-mobile/feature/signal-output-v2` (mobile — different repo)
2. Land backend PR first. Web + mobile PRs reference the backend PR URL in their description.
3. Web and mobile PRs can land in parallel once the backend is merged and deployed.
4. Never land a frontend PR against an unmerged backend — the contract it's coded against can still change.

**Commit message:** include a `[cross-repo]` tag when the change is part of a multi-repo feature, so later `git log` reads make the relationship obvious.

## 6. What lives where — the 30-second map

| If you're changing... | Work in... | Uses Claude for... |
|---|---|---|
| Python business logic, new endpoint, schema | `gcp3/backend/` | Sonnet; `/deploy` skill |
| Web UI (Next.js routes, components) | `gcp3/frontend/` | Sonnet; Server Components by default |
| Mobile UI (Expo screens, components) | `gcp3-mobile/` | Sonnet; Clerk auth is Phase 2/3 |
| Contract / shared types | `gcp3/backend/schemas/` → regenerate | Opus if redesigning; Sonnet to mechanically apply |
| Infra (Cloud Run, Vercel, Cloud Build) | `gcp3/` root | Sonnet; always `/post-deploy-verify` after |
| Scheduler jobs, secrets | `gcp3/` root | Opus — these are the gotcha surface |

## 7. General Claude best practices (both repos)

- **Read before you write.** Ask Claude to read the current file before proposing edits. Claude sometimes invents APIs that don't exist in your codebase.
- **Don't accept aspirational summaries.** If Claude says "wired in," verify with a grep. The Section A/B doc claimed mobile components were "wired in" — they weren't. Trust tool output, not prose.
- **Delete the generated `MULTI_REPO_*` docs when they go stale.** (Actually — no. Per `CLAUDE.md`, archive them to `docs/archived/`. Never delete.)
- **One commit per logical unit.** Don't let Claude batch a schema change, a UI change, and a CI change into one commit.
- **Verify the diff before committing.** Claude's commit messages describe intent; `git diff --cached` shows reality.
- **Don't let Claude push without you.** `git push` is one of the few remote-visible actions. Always review locally first.

## 8. Testing across repos

- Backend: pytest against Firestore emulator + mocked Finnhub/Gemini
- Web: Playwright against a running backend (staging or local)
- Mobile: Expo Go on device, hitting staging backend

**Contract tests** (the missing layer): add a test in each frontend that fetches a sample payload from staging and validates it against the generated types. This catches backend drift *before* users do.

---

## 9. Failure modes to watch for

| Symptom | Cause | Fix |
|---|---|---|
| Component imports `../backend/schemas/X` and path doesn't exist | Hand-written TS mirror drifted or never existed | Regenerate from Pydantic; delete mirror |
| Endpoint returns 500 on new field | Backend deployed, scheduler/env var not updated | `/post-deploy-verify`; check all 5 scheduler jobs |
| Mobile works locally, breaks in build | Expo dependency conflict (see `de20264`) | Pin exact versions in `package.json`; run `npx expo-doctor` |
| Web shows stale data for hours | Next.js ISR cache (`revalidate: 3600`) masking a backend bug | Trigger revalidation or lower the cache window temporarily |
| Backend test passes, prod breaks | Integration against mocked Finnhub/Gemini; real API changed shape | Add smoke tests against real staging |

## 10. Ten rules to add to both CLAUDE.md files

Append these verbatim to both [gcp3/.claude/CLAUDE.md](../../gcp3/.claude/CLAUDE.md) and [gcp3-mobile/.claude/CLAUDE.md](../.claude/CLAUDE.md). They're identical on purpose — both Claudes need the same mental model of the boundary.

```markdown
## Cross-Repo Workflow Rules

This app spans two repos with separate GitHub remotes:
- `gcp3` (adamaslan/gcp3.git) — Python backend + Next.js web frontend
- `gcp3-mobile` (adamaslan/gcp-expo1.git) — Expo/React Native mobile

The following rules apply in both.

1. **Backend is the source of truth.** `gcp3/backend/schemas/` defines every shared type. Frontends follow. Never invent a frontend-only shape that duplicates a backend concept.

2. **No hand-maintained TS mirrors of Python types.** Generate TS from Pydantic via `datamodel-code-generator` → `openapi-typescript`. Generated files get an `// AUTOGENERATED — do not edit` header and are committed.

3. **Verify before claiming "wired in."** If you write that a component, endpoint, or feature is integrated, prove it with a grep or a running test. No aspirational summaries. The Section A/B doc is the cautionary example.

4. **Cross-repo features land backend-first.** Backend PR merges and deploys before web/mobile PRs merge. Frontend PRs reference the backend PR URL in their description.

5. **Match branch names across repos.** A multi-repo feature uses the same branch name in `gcp3` and `gcp3-mobile` — e.g. `feature/signal-output-v2` in both. Makes `git log` across the two obvious.

6. **Handoff docs live in the backend repo.** When a backend change requires frontend work, write a short spec in `gcp3/docs/handoff/{date}_{feature}.md` — new types, what each frontend does, how to verify. Frontend sessions read the handoff; no guessing across context boundaries.

7. **Never edit the other repo from this session.** If this session is in `gcp3-mobile`, don't propose edits to files in `gcp3/`. Flag the cross-repo dependency instead and let the user switch sessions. Mixing contexts produces wrong-stack suggestions.

8. **Contract changes require three checks.** Schema change in backend → regenerated TS committed to web → regenerated TS committed to mobile. A PR that touches a schema without all three is incomplete.

9. **Trust tool output over prose.** If a doc says "X exists" and grep can't find X, X doesn't exist. Update the doc.

10. **Archive, don't delete.** When docs, code, or config become obsolete, move them to `docs/archived/{YYYY-MM-DD}_{reason}/`. Deletion requires explicit user approval in chat. See existing archival policy in this file.
```
